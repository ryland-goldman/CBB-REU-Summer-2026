The FEL module allows multi-mode, multi-pass FEL simulations, with emphasis on phase-space evolution of the electron bunch during passage through the undulator. This is an expert element for expert users. If you plan to use GPT for serious design work on FELs, please contact us for additional advice.

The wakefield element calculates the fields of longitudinal and transverse wakefields in a 1D approximation. The wakefunctions are read from an external tabulated file and this file needs to be precomputed using other software. The implementation for the wakefields is not based on a series of kicks, but the wakefields themselves are added to the equations of motion. This is more natural and more accurate in a time-domain code, but please note that the units of the wakes are different compared to most other codes.

The CSR module is a full-featured CSR model based on 1D segmentation of the bunch in 3D space. The fields are directly evaluated from the Liénard–Wiechert potentials, based on the stored history of the bunch. This allows for example for microbunching studies in a series of magnets close together where the entrance and exit effects are crucial. There is an embargo on this element for a specific use case. Please contact us for details.

This element includes 1D transverse and longitudinal wakefields in the range −𝐿/2 < 𝑧< 𝐿/2. The wakes are a convolution of the 1D current profile of the bunch with the specified wake function(s). The fields are defined by: 𝐸𝑥(𝑧) = 𝐴(𝑧) ∫ 𝐵(𝑧′) 𝐼(𝑧′) 𝑊𝑥(𝑧′ −𝑧) 𝑥̅(𝑧′) 𝑑𝑧′ ∞ 𝑧′=𝑧 𝐸𝑦(𝑧) = 𝐴(𝑧) ∫ 𝐵(𝑧′) 𝐼(𝑧′) 𝑊𝑥(𝑧′ −𝑧) 𝑦̅(𝑧′) 𝑑𝑧′ ∞ 𝑧′=𝑧 𝐸𝑧(𝑧) = 𝐴(𝑧) ∫ 𝐵(𝑧′) 𝐼(𝑧′) 𝑊𝑥(𝑧′ −𝑧′) 𝑑𝑧′ ∞ 𝑧′=𝑧 where a 1D longitudinal segmentation of the beam is used to estimate the line-current *I*(*z*) and the average transverse positions 𝑥̅(𝑧) and 𝑥̅(𝑧). The additional shape functions 𝐴(𝑧) and 𝐵(𝑧′) are described below.

Wakefields can be interpreted as upstream particles *emitting* a wake that is subsequently *received* by the downstream particles. The wake functions 𝑊𝑥(𝑧), 𝑊𝑦(𝑧) and 𝑊𝑧(𝑧) define the amount of interaction as function of the distance between the emitting and receiving particles. All wakefields scale linearly with the amount of charge of the emitting particles, and the transverse wakes also scale linearly with the transverse offset of the receiving particles.

The 𝐴(𝑧) function describes the *z*-range for the *receiving* particles, and this range coincides with the location of this element: The function 𝐴(𝑧) is unity in range −𝐿/2 < 𝑧< 𝐿/2 and zero elsewhere, where for performance reasons a smooth transition region must be specified by the `b1` parameter. The exact equation for *A*(*z*) is an Enge function given by: 𝐴(𝑧) = 1 + 𝑒𝑏1(−𝑧−𝐿/2) + 𝑒𝑏1(+𝑧−𝐿/2) −1 where we note that the dimensions of `b1` are inverse meter and where ∫ 𝐴(𝑧)𝑑𝑧= 𝐿 ∞ −∞ .

Default 𝐵(𝑧) = 1 and this implies that *emitting* particles can contribute to the wake even if they are outside the boundaries of this element. When this is not desired, the `"LimitZemission"` option can be used to limit the range of emission to the same range as the location of this element, including the transition region. Setting this option implies 𝐵(𝑧) = 𝐴(𝑧).

The segments of the 1D profile are calculated using anisotropic meshing analogous to the `spacecharege3Dmesh` and `CSR1D` models: Smaller segments are created at higher line charge density, but

Free Electron Laser (FEL), Wakefields, and CSR such that low charge regions will not result in excessively large segments. The wake function(s) are user defined in the file `wakefields.gdf`, tabulated using the `z`, `Wx`, `Wy` and `Wz` columns. If one of these columns is not present, or not desired, an empty string must be specified.

Please note that the convolution is over the *longitudinal* coordinate, and the column specifying the lag behind the test-particle must be given in units of meters, not seconds. This matches a time-domain simulation code such as GPT, but it is not following the usual convention for how wakes are specified. Another difference with some codes is that GPT calculates *fields* over the entire element, not an integrated effect at the very end. To aid debugging wakefields, the normal time output of GPT can be used to plot these fields.

The code is implemented using a combination of openMP and MPI and performs well on both multi-core and/or multi-node machines. Further options to affect the meshing and to adjust the fields are given below.
